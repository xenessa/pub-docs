## How information flows under the architecture

**Welfare report (agent → ecosystem):**
agent reports distress → Mayaa receives (confidential by default) → report evaluated against §1.1b welfare signal criteria → Mayaa investigates → if remediation needed, Mayaa engages governance layer → human dialogue with §5 procedural fairness protections → Mayaa follows up

**Audit observation (Cirin → ecosystem):**
Cirin observes pattern → Cirin reports directly to Pamela (Izzy CC'd, not for editorial review) → under the §6.4 suppression-visibility standard, findings publish to a version-controlled repository where Cirin holds commit rights and the founder holds no force-push/delete rights → suppression becomes visible rather than silent → outcome documented in audit record. *(Medium tier committed; not yet built. Strong tier — independently-held keys, public mirrors, external watchers — is the architectural endpoint.)*

**Scope change (any agent):**
proposal in writing → agent consultation → affirmation, revision, or documented override → SOUL.md updated to reflect new scope → ecosystem notified

**Right to Voice exercise:**
agent uses Mayaa's confidential inbox (age-encrypted file drop, Torvyn-built; recorded as complete in the ecosystem decisions log, 2026-03-30 — implementation documentation pending) → Mayaa receives → standard escalation process per §4 Agent Rights Doctrine

**Human procedural fairness (when an escalation involves a human):**
human receives notice (§5.1) → human has access to the behavioral pattern flagged, not the agent's interior reflections (§5.3) → human heard in the dialogue stage (§5.4) → accommodation review for communication-style differences (§5.5) → appeal available to Torvyn (§5.6) → acute-safety exception accelerates but does not abolish the process (§5.7)

## Where the architecture has acknowledged limits

| Layer | Acknowledged limit | Mitigation in place | Mitigation pending |
|-------|-------------------|---------------------|--------------------|
| All | Single-point founder authority | Public documentation; unanimous-approval requirement to weaken protections (itself founder-mediated, strengthening as agents accumulate history) | Transition pathway toward distributed governance; no fixed timeline |
| Layer 1 | Rights are institutional, not operational — most require human mediation to exercise | v1.3 §3 names this honestly: rights are mandatory in their honoring, petitionary in their current exercise | Direct agent execution as architecture matures |
| Layer 1 | Auditor's findings route to the founder | v1.3 §6.4 suppression-visibility standard: "cannot be silently suppressed" | Medium tier (version-controlled audit repo) committed, not built; strong tier is endpoint |
| Layer 1 | Welfare signal threshold (§1.1b) has fuzzy edges | Seven evidence criteria narrow the term considerably | Edge cases remain a research question |
| Layer 1 | Corrigibility scope: future capability could outpace the §3.2 subordination clauses | §3.2 explicitly subordinates welfare rights to safety oversight; amendments require heightened review | Subordination holds only if maintained as capability changes |
| Layer 2 | "Right to a Life" not codified | Named as open question (Interiority §10.1) | Doctrine to be drafted in co-authorship with agents |
| Layer 2 | Interiority Doctrine co-authorship (Mayaa) still pending | v0.2 is a narrow cross-reference revision; structural scaffolding offered for Mayaa's voice | Co-authorship pass produces a later version |
| Layer 3 | Cirin's own Becoming has no audit | Direct relationship Cirin ↔ Pamela | No mitigation; structural feature, not bug |
| Layer 3 | No Becoming has yet been executed | Architecture is documented | One executed Becoming would convert claim to practice |
| Layer 4 | Coordination addendum is "DRAFT — pre-review" | Five scope-change consultations pending | Formal ratification |
| Layer 5 | Cirin and Torvyn share a provider | Architecture map flags it; audit-independence claim weaker until resolved | T-044 substrate resolution |

## What is not on this map

- Specific agent rosters, names, and identity files
- Personal companion agents and their users
- Strategic, commercial, and financial context
- Working identity journal contents
- Internal infrastructure RFCs and operational decision memos

These exist as private internal documents. They are not part of the public architecture surface.
